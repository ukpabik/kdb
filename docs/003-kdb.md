# KDB
## Design Doc #3 – High-Performance LSM-Tree Persistent Store

**Abstract:**
KDB is a performant, thread-safe, persistent key-value storage engine, capable of handling arbitrary data using byte-based keys and values. This document outlines the V2 architecture: a fully asynchronous **Log-Structured Merge-Tree (LSM-Tree)**. The system utilizes double-buffered memory tables, Write-Ahead Logging (WAL) via direct I/O, thread-safe background flushing, K-Way merge compaction, and Bloom Filters. By leveraging `ByteBuffer` boundaries, sparse indexing, and `ReadWriteLock` concurrency, the system guarantees strict API contracts, extreme write throughput, and highly optimized, sub-millisecond reads with zero data loss.

---

## 1. Objectives

### Goals
* **High Concurrency & Low Latency:** Support high-throughput read/write operations (p99 < 1ms) without blocking foreground threads during disk I/O.
* **LSM-Tree Implementation:** Efficiently coordinate volatile memory (`MemTable`), persistent durability (`WriteAheadLog`), and immutable disk-based searches (`SSTable`).
* **Read Optimization:** Drastically reduce read amplification for non-existent keys using In-Memory Bloom Filters and an internal Sparse Index structure.
* **Space Reclamation:** Implement background K-Way merge compaction to asynchronously reconcile deleted (`TOMBSTONE`) and duplicate records.
* **Crash Resiliency:** Guarantee zero data loss across JVM or hardware crashes via exact sequential WAL replay and safe file boundaries.

### Non-Goals
* Making this key-value store distributed (reserved for Phase 4: Network Protocol).
* Opting for multiple implementations using different types for the KV store (strict `ByteBuffer` to `byte[]` architecture).

## 2. System Architecture

The KDB storage engine is fundamentally divided into a fast in-memory execution path and an asynchronous, durable on-disk path.

### 2.1 The Write Path (Puts and Deletes)
1. **Write-Ahead Log (WAL):** Before any data modifies memory, the operation (`PUT` or `DELETE`) is appended sequentially to the active `.log` file on disk via a `FileChannel`. This ensures data survives a sudden JVM crash.
2. **MemTable:** Once secured in the WAL, the record is inserted into the active `MemTable`, which is backed by a highly concurrent, non-blocking `ConcurrentSkipListMap`.
3. **Tombstones:** Deletions (`remove`) are treated identically to writes. Instead of erasing the key, a special 0-byte `TOMBSTONE` payload is appended. This "shadows" any older values of that key residing deeper in the disk files.

### 2.2 The Flush Mechanism (Double-Buffering)
To prevent write stalls during high-volume ingestions, KDB implements a double-buffering flush rotation:
* When the `activeMemTable` reaches the `FLUSH_CAPACITY` (e.g., 64MB), the `ReadWriteLock` momentarily halts writers.
* The active table is swapped to the `inactiveMemTable`, and a fresh `MemTable` and `WAL` are instantly provisioned for new writes. The write lock is released.
* A background `ExecutorService` thread safely drains the immutable `inactiveMemTable` into an `SSTableWriter`, flushing the sorted entries sequentially to an `.sst` file on disk, after which the old WAL is discarded.

### 2.3 The Read Path
To maintain sub-4 microsecond read latency, reads cascade through the system tiers in order of chronological authority:
1. **Active MemTable:** Fast O(log N) lookup in volatile memory.
2. **Inactive MemTable:** Fast O(log N) lookup in the currently flushing buffer.
3. **In-Memory Bloom Filters:** If the key is not in memory, KDB checks the Bloom Filter associated with each `SSTable`. If the filter returns false, the disk read is mathematically short-circuited.
4. **Sparse Index:** If the Bloom Filter returns true, KDB consults the SSTable's Sparse Index (kept in memory), which contains 1 key pointer for every 100 keys (the `INDEX_SEGMENT`). This locates the exact byte offset of the data block.
5. **Disk Seek:** KDB uses `SafeReadWrite` and `MappedByteBuffer` to perform a targeted, localized sequential scan inside the `SSTable` block to fetch the exact payload.

### 2.4 Asynchronous K-Way Compaction
To regulate disk growth and clear obsolete Tombstones, the `CompactionManager` triggers a background thread when the `COMPACTION_CAPACITY` threshold is met.
* **Min-Heap Merge:** It isolates the target SSTables and opens sequential `SSTableIterator` streams into a `PriorityQueue` (Min-Heap).
* **Ghost Pop Deduplication:** The custom comparator sorts lexicographically by key, resolving identical collisions by falling back to the sequence number. The newest version wins, and the "Ghost Pop Guard" discards older duplicate frames without writing them.
* **Atomic Handover:** The consolidated data is streamed to a temporary descriptor (`.tmp`). Once the sparse index and footer are written, `Files.move` atomically swaps the `.tmp` to an `.sst` file, seamlessly replacing the old tables in the `SSTableManager` with zero downtime.

## 3. Storage Layouts and Serialization

KDB uses tight, predictable byte layouts to maximize channel throughput and prevent buffer overflows.

### 3.1 Write-Ahead Log (WAL) Format
The WAL is an append-only sequence of binary packets:

    [1 byte: OpCode (0 = PUT, 1 = DELETE)]
    [4 bytes: Key Size (int)]
    [N bytes: Raw Key Data]
    [4 bytes: Value Size (int)] (Omitted if OpCode is DELETE)
    [M bytes: Raw Value Data]

### 3.2 SSTable (.sst) Format
SSTables are self-contained, single-file architectures. They do not rely on separate `.index` files, ensuring file integrity.

    === DATA BLOCKS ===
    [4 bytes: Key Size] [N bytes: Key Data] [4 bytes: Value Size] [M bytes: Value Data]
    ... (Repeats sequentially for all entries)

    === SPARSE INDEX ===
    [4 bytes: Key Size] [N bytes: Key Data] [8 bytes: Absolute Physical Offset]
    ... (Repeats for every 100th key)

    === BLOOM FILTER ===
    [Guava Bloom Filter serialized binary output]

    === FOOTER ===
    [8 bytes: Index Offset] 
    [8 bytes: Index Size] 
    [8 bytes: Bloom Filter Offset] 
    [4 bytes: MAGIC_NUMBER (Verification signature)]

## 4. Crash Recovery

Upon initialization, the `PersistentStore` delegates to the `WriteAheadLog` to rebuild state.
* KDB scans the data directory for `.log` files.
* Using a `FileChannel`, it bypasses the JVM heap overhead and reads the binary packets sequentially.
* It replays the `OpCodes` directly into a new `MemTable`.
* If a file ends prematurely (e.g., a power cut mid-write), the `SafeReadWrite` system detects the `BufferUnderflow` or `EOF`, discards the partial frame, and seamlessly completes the boot process.

## 5. Constraints & Scalability

### Constraints
* **Memory Limits:** * `FLUSH_CAPACITY`: Capped (e.g., 64MB) to control JVM garbage collection pressure.
  * `COMPACTION_CAPACITY`: 6 SSTable files before triggering a background merge.
* **Key/Value Limits:** Engineered specifically for byte-based records. Overhead is mitigated by direct `ByteBuffer.duplicate()` sharing rather than array copies.
* **JVM Lifecycle:** `PersistentStore.close()` must be invoked to gracefully terminate `ExecutorService` thread pools, flush active logs, and release OS-level file locks.

### Scalability
* System scales heavily toward write-intensive workloads.
* The `ConcurrentSkipListMap` and Thread Pools prevent `ConcurrentModificationException` and I/O bottlenecks.
* Memory usage remains remarkably stable due to `KVPair` streaming and boundary pointers during compaction merges.

## 6. Performance Benchmarks

The robust engineering architecture translates directly into world-class latency parameters. The following benchmark reflects a 1,000,000 entry load (16-byte keys, 100-byte values):

| Operation | Latency (micros/op) | Throughput (MB/s) |
| :--- | :--- | :--- |
| **fillrandom** (Writes) | 3.978 µs/op | ~27.8 MB/s |
| **overwrite** (Updates) | 3.959 µs/op | ~27.9 MB/s |
| **readrandom** (Reads) | 3.555 µs/op | ~31.1 MB/s |

**Analysis:**
* **Sub-4 Microsecond Writes:** Validates that the active WAL FileChannel and lock-free MemTable completely absorb disk latency.
* **Reads outperforming Writes:** Proves that the In-Memory Bloom Filters and Sparse Indexes are successfully short-circuiting disk seeks, achieving near O(1) read performance despite files resting on physical drives.
* **Zero Data Loss:** 1,000,000 keys were written and 1,000,000 keys were successfully retrieved during live compaction, confirming thread-isolation and atomic file guarantees.

## 7. Future Considerations
* **Networking Layer:** Implement an asynchronous TCP/IP interface (Java NIO or Netty) to allow external remote clients to interface with the database.
* **Distributed Clustering:** Introduce a consensus protocol (e.g., Raft) and consistent hashing to partition data across a multi-node cluster.
* **LRU Block Caching:** Implement an in-memory Block Cache to retain recently read chunks of SSTables in memory, further reducing I/O for frequently accessed (hot) keys.
