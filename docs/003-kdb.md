# KDB
## Design Doc #3 – High-Performance LSM-Tree Persistent Store

**Abstract:**
KDB requires a performant, thread-safe persistent key-value store capable of handling arbitrary data using byte-based keys and values. This document outlines the upgraded V2 architecture, implementing a fully asynchronous Log-Structured Merge-tree (LSM-Tree). The system now features non-blocking double-buffered memory tables, thread-safe background flushing, K-Way merge compaction, and Guava Bloom Filters. By leveraging `ByteBuffer` and `ReadWriteLock` concurrency, the system guarantees strong API contracts, high-throughput writes, and highly optimized reads.

---

## 1. Objectives

### Goals
* **Concurrency:** Support high-throughput read/write operations without blocking foreground threads during disk I/O.
* **LSM-Tree Implementation:** Efficiently coordinate volatile memory (`MemTable`), persistent `WriteAheadLog` (WAL), and immutable disk-based `SSTable`s.
* **Read Optimization:** Drastically reduce read amplification for non-existent keys using Bloom Filters and Sparse Indexing.
* **Space Reclamation:** Implement background K-Way merge compaction to reconcile deleted (`TOMBSTONE`) and duplicate records.
* **Crash Resiliency:** Guarantee zero data loss across JVM or system crashes via sequential WAL replay.

### Non-Goals
* Making this key-value store distributed (reserved for future iterations).
* Opting for multiple implementations using different types for the KV store.

---

## 2. Proposed Design

### Main Data Structures Used
* `Store<K, V>` (Interface)
* `PersistentStore` (Coordinator utilizing `ReentrantReadWriteLock`)
* `MemTable` (Volatile tier backed by `ConcurrentSkipListMap` and `AtomicLong` for precise byte tracking)
* `WriteAheadLog` (Append-only recovery tier)
* `SSTable`, `SSTableWriter`, `SSTableManager` (Immutable on-disk tier with sparse indexing)
* `SSTableIterator` (Sequential streaming engine for background processes)
* `CompactionManager` (Min-heap based garbage collection)

### API Contracts
**Contracts:** Key and Value must never be null.

* `Optional<byte[]> get(ByteBuffer key)`
    * Queries the Active MemTable → Inactive MemTable → SSTables (newest to oldest).
    * Returns an empty optional if the key is not found or if a `TOMBSTONE` is encountered.
* `void put(ByteBuffer key, byte[] value)`
    * Appends to the WAL, then applies to the Active MemTable.
    * Triggers an asynchronous flush if the Active MemTable crosses the 4MB threshold.
* `Optional<byte[]> remove(ByteBuffer key)`
    * Maps the given key to a zero-byte `TOMBSTONE` array.
    * Returns the previous value if it existed in the active memory tier.

## 3. Architecture & Mechanics

### Double-Buffered Flushing
When the `activeMemTable` reaches its 4MB capacity, it is atomically rotated to `inactiveMemTable`, and a fresh MemTable is provisioned. A background `ExecutorService` sequentially flushes the inactive table to an `SSTable` on disk, then drops the old WAL. This ensures incoming write traffic is never paused by disk I/O.

### On-Disk SSTable Layout
The `.sst` files are optimized for minimal memory footprint:
1. **Data Blocks:** Sequential `[Key Size][Key][Value Size][Value]` blocks.
2. **Sparse Index:** An offset map taking a snapshot every 100 keys (`INDEX_SEGMENT`).
3. **Bloom Filter:** A serialized Guava `BloomFilter` (1% false positive probability).
4. **Footer:** A 28-byte block tracking index offsets, bloom offsets, and a validation Magic Number (`0x4B444249`).

### Thread-Safe K-Way Compaction
Triggered when the file count hits `COMPACTION_CAPACITY` (6 files). A background thread uses a `PriorityQueue` (Min-Heap) to merge iterators from all active SSTables. It resolves key collisions by prioritizing higher sequence numbers. After writing the compacted `.tmp` file, it atomically replaces the obsolete source files via `SSTableManager`.

## 4. Design Tradeoffs and Alternatives

### Tradeoff #1: Double-Buffered MemTables vs. Blocking Writes
* **Pros:** Standard writes take microseconds. The system achieves massive throughput by offloading disk I/O to a background thread.
* **Cons:** Temporary memory footprint doubles (up to 8MB) during a flush cycle.
* **Alternatives:** Blocking writes while flushing (too slow).

### Tradeoff #2: Bloom Filters vs. Raw Disk Reads
* **Pros:** Instantly rejects queries for keys that do not exist, completely bypassing slow file I/O operations and read amplification.
* **Cons:** Adds CPU overhead for hashing and increases the disk footprint of `.sst` files to store the serialized filter arrays.

### Tradeoff #3: Sparse Index vs. Dense Index
* **Pros:** Keeps memory usage extremely low. Only 1 in 100 keys are loaded into RAM via the `TreeMap`.
* **Cons:** Requires sequential disk scanning within the 100-key segment block to locate exact records.
* **Alternatives:** Dense index (mapping every key). Unscalable memory usage for large datasets.

### Tradeoff #4: LSM Tree vs. B-Tree
* **Pros:** Transform random writes into sequential appends, maximizing write IOPS and extending SSD lifespan.
* **Cons:** Read queries are fundamentally slower due to checking multiple tiers (MemTable, Inactive MemTable, multiple SSTables).

## 5. Constraints & Scalability

### Constraints
* **Memory Limits:** * `FLUSH_CAPACITY`: 4,000,000 bytes (~4 MB) per MemTable.
    * `COMPACTION_CAPACITY`: 6 SSTable files before triggering a background K-Way merge.
* **Data Limits:**
    * Key: Recommended 50-byte maximum.
    * Value: 1 MB maximum.
* **JVM Lifecycle:** `PersistentStore.close()` must be invoked to shut down background `ExecutorService` threads gracefully and release file locks.

### Scalability
* System scales heavily toward write-intensive workloads.
* Background threads prevent `ConcurrentModificationException` and I/O bottlenecks.

## 6. Testing Capabilities
* **Unit Testing:** `MemTable` size calculations, tombstones, and concurrent skip list operations.
* **Integration Testing:** `PersistentStore` operations utilizing `ReadWriteLock`.
* **Crash Recovery:** Simulating process death and validating proper reconstruction of the MemTable from the un-flushed `.log` (WAL) files.
* **Concurrency Testing:** High-volume multithreaded puts/gets to ensure the atomic flush rotation does not drop data.
* **Compaction Testing:** Verifying deduplication and `TOMBSTONE` garbage collection.

---

## 7. Future Considerations
* **Distributed Architecture:** The storage engine is now highly optimized for a single-node setup. The next iteration should implement a consensus protocol (like Raft) and consistent hashing to partition data across a cluster of nodes.
* **Network Protocol:** Implement a lightweight TCP/IP interface (similar to RESP for Redis) to allow remote clients to interface with the database.
